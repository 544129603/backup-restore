## 3. 关键用例证据

- Repository fake/client：健康检查设置 Ready 和 finalizer。
- Policy：同一 `policyUID/scheduledTime` 连续 Reconcile 只创建一个 Task，且冻结 Scope snapshot。
- BackupTask：取消后进入终态 Cancelled 并记录完成时间。
- BackupRecord：只有 `.done` 与 manifest 校验通过才变为 Available。
- SFTP：真实 SSH 握手、密码认证、known_hosts 校验和 SFTP 子系统读写均由进程内服务器验证，不是 mock 返回值。
- Encryption：超过一个 chunk 的数据加解密一致；单字节篡改被 GCM authentication 拒绝。
- Archive：稳定顺序/时间戳，拒绝符号链接与 `../` 解包路径。
- Sanitizer：删除 status、UID/resourceVersion/managedFields/finalizer/ownerReference，以及 Service/PVC/PV/Job 不可移植字段。

## 4. 未执行或受环境限制的项目

- 当前桌面没有可用 Kubernetes 集群和真实 CSI Driver，因此没有声称通过真实 VolumeSnapshot/PVC Bound E2E。
- Windows 自带的 MinGW `cc1.exe` 不支持 64 位，`go test -race` 在编译 `runtime/cgo` 时失败；普通测试、Vet、Build 均通过，Race 已作为 Linux CI 强制步骤。
- 当前没有企业 SFTP、故障注入代理和大规模集群，因此尚未完成网络抖动、磁盘满、10 万对象和 100 PVC 的生产压测。
- Kustomize 的 `kubectl apply --dry-run=client` 能解析所有内置对象，但本机未安装 cert-manager CRD，Certificate/Issuer 的 client-side mapping 检查按预期失败；Helm/Kustomize 渲染本身通过。
- 已提供 `test/e2e/run.ps1` 真实集群冒烟入口，覆盖 Helm 安装、七个 Cluster-scoped CRD、控制器 Ready、Admission 拒绝和 API 查询；当前没有一次性 Kubernetes 集群与可拉取镜像，因此只完成脚本语法验证，未将其列为已通过项。
- 浏览器平台页面和平台 REST API 属于上一阶段产品设计，本仓库交付的是 Operator 后端，不包含现有平台前端工程。

这些项目是发布门禁而不是已通过项，不能据此宣称 Production Ready。

## 5. 发布前建议门禁

1. 在目标 Kubernetes 1.28–1.32 与每个 CSI 厂商上跑真实 E2E：快照成功、超时、部分失败、原位/映射恢复、删除策略。
2. 用企业 SFTP 测试 host key 轮换、rename/statvfs 差异、上传中断和至少 100GiB 流式包。
3. 完成 1 万/10 万对象、100 PVC、3 并发备份和 1 并发恢复的 CPU/内存/临时盘压测。
4. 将 Reconcile 内长任务迁移为受限 Worker Job 后，再评审超大集群 Production Ready。
5. 替换 module path、镜像地址、正式 clusterRef/权限目录，并完成镜像签名、SBOM、漏洞扫描。
