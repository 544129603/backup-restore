# 10 部署与操作手册

## 1. 前置条件

- Kubernetes 1.28–1.32；研发验证使用 EnvTest Kubernetes 1.32.0。
- CSI 快照使用 `snapshot.storage.k8s.io/v1`；集群须已安装 snapshot-controller、CRD 和匹配的 `VolumeSnapshotClass`。
- 默认 Kustomize/Helm Webhook TLS 使用 cert-manager v1 API。
- Local Repo 要求 Operator Pod 已挂载对应目录；SFTP Repo 要求网络、账号和 `known_hosts` 已准备。
- 生产必须设置唯一 `clusterRef`，并保证平台 API 写入的 `spec.clusterRef` 与它一致。

## 2. 构建

```powershell
go mod download
make generate manifests
go test ./...
go build -trimpath -o bin/manager.exe ./cmd/manager
docker build -t registry.example/backup-restore-operator:0.1.0 .
```

生成物包括 7 个 CRD、ClusterRole 和 Mutating/ValidatingWebhookConfiguration。CI 会在生成后执行 `git diff --exit-code`，防止代码和清单漂移。

## 3. Helm 安装

```powershell
helm upgrade --install backup-restore charts/backup-restore-operator `
  --namespace backup-system --create-namespace `
  --set clusterRef=cluster-a `
  --set image.repository=registry.example/backup-restore-operator `
  --set image.tag=0.1.0
```

Local hostPath 额外参数：

```powershell
helm upgrade --install backup-restore charts/backup-restore-operator `
  -n backup-system --create-namespace `
  --set clusterRef=cluster-a `
  --set localRepositoryMount.enabled=true `
  --set localRepositoryMount.hostPath=/var/lib/backup-repository `
  --set localRepositoryMount.mountPath=/repository `
  --set nodeSelector.kubernetes\.io/hostname=worker-backup-01
```

`BackupRepository.spec.local.path` 必须填 `/repository` 或其安全子目录。hostPath 与业务数据处于同一故障域，不能作为异地灾备。

## 4. Kustomize 安装

修改 `config/manager/manager.yaml` 的镜像和 `--cluster-ref` 后执行：

```powershell
kubectl apply -k config/default
kubectl -n backup-system rollout status deploy/backup-restore-operator
kubectl get crd | Select-String protection.platform.io
```

若不安装 cert-manager，不得应用默认 Webhook 清单；必须预创建 TLS Secret、注入 `caBundle`，或仅在本地开发时用 `--enable-webhooks=false`。

## 5. 凭据

```powershell
kubectl -n backup-system create secret generic sftp-primary `
  --from-literal=username='backup-user' `
  --from-literal=password='REDACTED' `
  --from-file=known_hosts=./known_hosts

# 精确 32 字节的 AES-256 key；Secret 应设置 immutable: true 并使用版本化名称轮换。
kubectl -n backup-system create secret generic backup-encryption-v1 `
  --from-file=key=./aes-256.key
```

不要提交包含真实 `stringData` 的 YAML，不要复用或覆盖旧版本加密 Secret；旧 Record 恢复依赖其冻结引用。

## 6. 操作流程

1. 创建 Repository，等待 `Ready`：`kubectl wait --for=jsonpath='{.status.phase}'=Ready backuprepository/<name> --timeout=60s`。
2. 创建 Scope，检查 `status.preview` 和 `ScopeResolved` Condition。
3. 创建 Policy；`status.nextScheduleTime` 应出现。立即执行时创建独立 Manual BackupTask。
4. 查看 Task：`kubectl get backuptask -w`；失败查看 `status.errorCode`、`status.errors` 与 Operator 结构化日志。
5. 仅使用 `status.restorable=true` 的 Record 创建 RestoreTask。
6. 高风险恢复先使用 `dryRun: true`；确认冲突数量和 plan hash 后创建新的非 DryRun Task。

取消任务通过 Patch `spec.cancelRequested=true`。备份越过 `.done` 后取消会被安全延迟，直到 Record 生成结束；恢复取消不回滚已创建资源，`status.residualResources` 给出最多 200 个清理提示。

## 7. 删除 BackupRecord

三种模式必须明确选择：

```powershell
kubectl annotate backuprecord/<name> `
  protection.platform.io/delete-mode=RepositoryDataAndSnapshots `
  protection.platform.io/delete-confirmed=true --overwrite
kubectl delete backuprecord/<name>
```

- `RecordOnly`：仅删 CR，Repo 数据和快照保留，可能形成不可管理资产。
- `RepositoryData`：删除 CR 与包，保留快照。
- `RepositoryDataAndSnapshots`：删除 CR、包和快照。

带 `protected=true` 的 Record 还需平台管理员显式设置 `force-delete=true`。RetentionController 会自动写入模式与确认标记。

## 8. 故障处理

| 现象 | 检查 | 处理 |
|---|---|---|
| Repo Failed | `status.errorCode`、Secret key、known_hosts、网络 | 修复后等待健康检查或更新 CR 触发 Reconcile |
| `BR-REPO-NOSPACE-004` | `availableBytes` 与 minimumFreeBytes | 清理或扩容；不绕过容量门槛 |
| SnapshotUnsupported | SC provisioner 与 VSC driver/default | 指定唯一匹配 SnapshotClass |
| Backup 卡在快照 | VolumeSnapshot `readyToUse/error` | 修复 CSI；到 Task timeout 后失败，创建 Retry Task |
| Record Broken | `.done`、manifest、远端对象 | 不允许恢复；从源重新备份 |
| Restore Conflict | DryRun plan、冲突策略 | 优先 Skip；Overwrite/Rename 逐资源配置 |
| PVC 未 Bound | 目标 SC、快照、CSI events | 修复存储；验证阶段持续等待直到 Task timeout |

## 9. 升级与卸载

升级前先备份 CR 和 Secret 元数据，执行 Helm upgrade，观察 CRD conversion/manager 日志。`v1alpha1` 不允许直接删除字段；包格式通过 `formatVersion` 管理。

卸载 Operator 不应删除 CRD；先保留所有 Record。只有确认不再恢复时，按三模式清理 Record，再手工删除 CRD。删除 Policy/Task/RestoreTask 不会级联删除 Record。
