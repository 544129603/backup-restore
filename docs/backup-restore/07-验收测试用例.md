# 容器平台备份与恢复插件：验收测试用例

## 1. 测试基线

### 1.1 环境与数据

- Kubernetes：认证矩阵内两个 minor（建议 1.30、1.34），安装 `snapshot.storage.k8s.io/v1` CRD、snapshot-controller；一个支持快照的 CSI StorageClass/VSC，一个不支持快照的 StorageClass。
- Namespace：`project-a-ns1`、`project-a-ns2`、`project-b-ns1`；项目 A/B 用户与平台/集群管理员。
- 每个项目包含 Deployment、StatefulSet、Service、ConfigMap、Secret、ServiceAccount、RBAC、PVC；含标签 `app=demo` 与 `backup=disabled` 的对象。
- Repo：一个 hostPath 固定节点、一个 Bound RWX PVC、一个可用 SFTP、一个错误凭据 SFTP、一个受配额/低空间 Repo。
- 可注入：controller kill、worker kill、API 429、SFTP 断连/篡改、CSI timeout、admission rejection、Operator 升级。

验收输出必须保存：CR YAML/status、Event、脱敏日志、Repo 目录清单、VolumeSnapshot/Content、API request/response、审计日志。所有“最终”条件默认在对应 timeout 内通过轮询验证，不以 UI toast 代替后端事实。

## 2. Repository 验收

| 用例 | Given | When | Then |
|---|---|---|---|
| AT-REPO-001 创建 Local hostPath Repo | 节点 `worker-1` 上 `/var/lib/backup` 可由 UID 1000 读写，空间>10Gi | 平台管理员创建 HostPath Repo，指定 worker-1，触发检查 | Repo phase=Ready；capabilities read/write/delete/atomicRename=true；测试文件被清理；执行 Job 固定 worker-1；重调度 Operator Pod 不影响 Repo |
| AT-REPO-002 Local 路径权限失败 | 目录只读或 UID 不匹配 | 创建/检查 Repo | phase=Unavailable；errorCode=`BR-REPO-PERMISSION-007`；日志不泄漏宿主其他目录；不可被新策略启用 |
| AT-REPO-003 Local PVC Repo | backup-system 的 RWX PVC Bound | 创建 PVC mode Repo 并检查 | Worker 挂载 `/repository/<subPath>`；读写/rename/delete 成功；status.resolved.pvcUID 正确；Repo Ready |
| AT-REPO-004 Operator/Worker 漂移 | HostPath Repo 绑定 worker-1，controller 位于 worker-2 | 创建手动 Task | Controller 可在 worker-2；backup worker 必须在 worker-1；若 worker-1 NotReady，Repo/Task 给出依赖不可用而不写其他节点同名路径 |
| AT-REPO-005 创建 SFTP password Repo | known_hosts 正确、Secret 包含 username/password | 创建并检查 | 依次通过 host key/auth/path/RW/rename/capacity；Repo Ready；SFTP 目录无健康检查残留；CR/log/API 无密码 |
| AT-REPO-006 创建 SFTP private-key Repo | 加密私钥、passphrase、known_hosts 均在 Secret | 创建并检查 | 认证成功；Secret data 不出现在 CR/status/Event/log；Secret RV 更新后自动重检 |
| AT-REPO-007 Repo 认证失败 | 密码错误 | 检查 SFTP Repo | phase=Unavailable；`BR-REPO-AUTH-002` retryable=false；UI 提示更新凭据；更新 Secret 后无需重建 Repo 即变 Ready |
| AT-REPO-008 Host key 不匹配 | 服务端 key 与 known_hosts 不同 | 检查 Repo | 连接被拒；`BR-REPO-HOSTKEY-003`；系统不自动接受 key；审计记录变更 known_hosts 的操作者 |
| AT-REPO-009 Repo 容量不足 | 可用空间低于 minFreeBytes/预计包+reserve | 启动备份 | Repo Degraded；Task 在 Preparing 失败，code=`BR-REPO-CAPACITY-006`；无快照/上传副作用或已创建副作用被清理 |
| AT-REPO-010 Repo 删除保护 | Repo 被 enabled Policy 和 1 个 Record 引用 | 请求 Safe 删除 | API 409/422 `BR-REPO-INUSE-005`，列出引用计数；Repo 未进入 deletionTimestamp；历史副本不受影响 |

## 3. Policy Selection 与快照验收

| 用例 | Given | When | Then |
|---|---|---|---|
| AT-SELECTION-001 Namespace 范围备份 | Policy selection include ns1/ns2，exclude ns2 | 预览并执行 | 预览与 Record 仅含 ns1；selectionHash 相同；不得出现 ns2 或 project-b 对象 |
| AT-SELECTION-002 Cluster 范围备份 | 平台管理员、selection.mode=Cluster、includeClusterResources=true | 执行 | 包含允许的 cluster-scoped 与 Namespace 对象；默认排除 Node/Event/Lease/插件自身 CR；source.selectionMode=Cluster |
| AT-SELECTION-003 资源类型过滤 | include deployments.apps,configmaps；exclude configmaps | 执行 | exclude 优先，归档仅含 Deployment（及明确的必需依赖策略项）；index 给出每类包含/排除计数 |
| AT-SELECTION-004 标签过滤 | labelSelector `app=demo`，存在匹配/不匹配对象 | 预览和执行 | 仅匹配对象进入 index；无标签的隐式依赖不被偷偷加入，除非 index 明示 dependency autoInclude=true |
| AT-SELECTION-005 默认运行时清洗 | 源对象含 uid/resourceVersion/managedFields/status/finalizer | 备份并解密检查归档 | 归档 manifest 不含这些字段；index 另存 source UID/owner logical ref；Secret data 仅在加密包中且不出日志 |
| AT-SELECTION-006 项目越权 Namespace | 项目 A 管理员提交 include project-b-ns1 | API/直接 CRD create | 平台 API 返回 403；绕过 API 的直接 create 被 RBAC 或 Webhook 拒绝；不得泄漏 project-b 对象计数 |
| AT-SNAPSHOT-001 PVC 快照成功 | CSI PVC/VSC driver 匹配 | 执行含 PVC 备份 | VS 确定名称；readyToUse=true；VSC handle 非空；Task succeededSnapshots=1；Record snapshotCount=1 且 Available |
| AT-SNAPSHOT-002 部分 PVC 快照失败 | 两 PVC，一支持一超时；failurePolicy=Continue | 执行 | Task=PartiallyFailed；成功快照与资源包完整；Record=SnapshotMissing，missing=1；可选择仅成功 PVC/元数据恢复 |
| AT-SNAPSHOT-003 快照失败 Fail | 同上但 failurePolicy=Fail | 执行 | Task=Failed；不生成可用 Record；未提交受管快照按 lifecycle 清理或列为 orphan；无 `.done` |
| AT-SNAPSHOT-004 不支持 StorageClass | NFS/in-tree PVC 无 CSI snapshot | 预览 Policy selection | `BR-SNAPSHOT-NOTSUPPORTED-001` 指向 PVC/SC；Fail 策略为 Blocking；用户可关闭 PVC 或排除该 PVC |
| AT-SNAPSHOT-005 生命周期 Retain | Record lifecycle=Retain | 删除 Record mode=Data | Repo 包删除，VS/VSC/后端快照保留；删除 mode=DataAndSnapshots 需另测并仅删受管资产 |

## 4. Policy 与 BackupTask 验收

| 用例 | Given | When | Then |
|---|---|---|---|
| AT-POLICY-001 定时策略触发 | Cron 在 2 分钟内到点、Repo Ready、selectionPreview 有效 | 到 scheduledTime | 仅生成一个 trigger=Schedule Task；annotation scheduledTime UTC 正确；Policy last/nextRun 更新；Task 固化 refs UID 与 selectionSnapshot |
| AT-POLICY-002 定时防重 | 对同一 policyUID/scheduledTime 并发触发两个 Reconcile | 并发创建 | Kubernetes 中只有一个 Task/idempotencyKey；另一次采纳 AlreadyExists；指标不重复计数 |
| AT-POLICY-003 禁止并发 | concurrency=Forbid，旧 Task Running，新时间到点 | 调度 | 不生成新 Task；Policy skippedRuns reason=ForbidActive；nextRun 正确；旧 Task不受影响 |
| AT-POLICY-004 Replace | concurrency=Replace，旧 Task Running | 到点 | 旧 Task cancelRequested=true 并进入 Cancelling；只有旧任务终态/宽限规则满足后新 Task 才执行；不并发写同 backupID |
| AT-POLICY-005 控制器重启补偿 | missed=RunOnce, deadline=1h；停 controller 跨过 3 个点 | 重启 | 只为窗口内最近 scheduledTime 创建 1 个 Task；早期点记 skipped；再次重启不重复 |
| AT-POLICY-006 DST/timezone | `America/New_York`，覆盖跳时/重复时模拟时钟 | 计算 future runs | 不存在本地时刻不执行；重复时刻以两个不同 UTC key 执行或被 Forbid 记录；UI/API 预览一致 |
| AT-POLICY-007 策略修改隔离 | 已有 Running Task，修改 selection/retention/timeout | 保存 | Running Task spec/hash 不变；下一个 Task 使用新 generation；历史 Record 不改内容；retention 是否重评符合 applyToExistingRecords |
| AT-POLICY-008 删除策略非级联 | Policy 有历史 Task/Record | 删除 Policy | Policy 删除；Task/Record 仍存在且可查询/恢复；无 ownerReference 级联 |
| AT-TASK-001 手动备份 | 用户有 backup.execute，Policy/Repo Ready | 仅提交 policyRef，并用同 Idempotency-Key 连续提交两次 | 返回同一个 Manual Task；selection/repository 被固化；仅一个包/快照集合/Record |
| AT-TASK-002 进度与步骤 | 100+对象、2 PVC | 执行期间查询 | phase 按合法顺序，progress 单调不倒退（重估须另字段解释）；status 至少每 2s/1%更新；日志/事件含 traceId |
| AT-TASK-003 备份取消 | Task 在 Collecting/Uploading，发送 cancel | 等待终态 | 30s 内 phase=Cancelling；最终 Cancelled；无 Available Record；staging 在 grace 内清理；残留快照明确列出 |
| AT-TASK-004 commit 后取消 | `.done` 已写且 GeneratingRecord | 发送 cancel | 返回 not cancellable 或 condition TooLateToCancel；Task 完成 Record；不得留下可用包却静默无 Record |
| AT-TASK-005 备份重试 | Task 因可重试网络错误 Failed | 调用 retry | 新 Task trigger=Retry,parentTaskRef 指向旧 Task；旧 status 不变；成功时生成新 backupID/Record；重试链可查 |
| AT-TASK-006 崩溃恢复 | 在 snapshot 创建后/status 写前 kill worker/controller | 恢复 controller | 通过 task UID/确定名称采纳已有 snapshot；无重复后端快照；60s 内恢复调谐；最终结果正确 |

## 5. BackupRecord 与完整性验收

| 用例 | Given | When | Then |
|---|---|---|---|
| AT-RECORD-001 生成 BackupRecord | 完整包已上传但 Record 尚未创建 | 让 controller reconcile | 仅在 `.done`、metadata/index、SHA-256、快照检查通过后创建 Record；Task.recordRef 指向其 UID；Record spec immutable |
| AT-RECORD-002 上传中断无可用副本 | SFTP 在 60% 断开直到 Task timeout | 检查 CR/Repo | 仅有 `.staging`/`.part`，无 `.done`；无 Available Record；Task Failed `BR-BACKUP-UPLOAD-001` |
| AT-RECORD-003 完整性校验成功 | Available Record | 触发 Full verify | 先 Verifying；最后 Available；filesExpected=filesVerified、checksumMatched=true、lastVerified 更新 |
| AT-RECORD-004 包被篡改 | 修改 `resources.tar.gz(.enc)` 一个字节 | Full verify | Record=Broken、`BR-BACKUP-CHECKSUM-002`、IntegrityVerified=False；恢复 API 拒绝；不自动删除证据 |
| AT-RECORD-005 Repo 暂不可用 | 阻断网络但不改数据 | verify | Record=RepoUnavailable 而不是 Broken；网络恢复后重验回 Available |
| AT-RECORD-006 Namespace 删除后记录仍存在 | 完成 ns1 备份并取得 Record | 删除源 Namespace | BackupRecord/包/快照不因 Namespace GC；仍能映射到新 Namespace 恢复；Record 无 namespaced ownerRef |
| AT-RECORD-007 三种删除模式 | 三份等价 Record | 分别 CROnly/Data/DataAndSnapshots 删除 | CROnly 只删 CR且报 orphan；Data 删 CR+包保留快照；DataAndSnapshots 再删确认归属的快照；每次均二次确认/审计 |

## 6. Restore 验收

| 用例 | Given | When | Then |
|---|---|---|---|
| AT-RESTORE-001 恢复前强制预检 | Available Record | 跳过 planHash 直接创建 RestoreTask | API/Webhook 422；不创建业务资源；运行有效 plan 后方可提交 |
| AT-RESTORE-002 原位恢复 | 源 Namespace 存在，删除其中 Deployment/ConfigMap，Record Available | Original + Skip 提交 | 缺失对象创建；已存在对象 Skip；Task Completed；对象 runtime 字段为目标新值；Record restore count+1 |
| AT-RESTORE-003 新 Namespace 恢复 | 目标 `app-dr` 不存在 | map `app-prod→app-dr` 并恢复 | 先创建 Namespace，再 PVC，再 Workload；所有 namespaced 引用重写；源 Namespace 不被修改 |
| AT-RESTORE-004 多 Namespace 映射 | 两源 Namespace 映射到两个新目标 | 恢复 | 对象落入对应目标；不允许未经明确授权将两个源映射到同一目标；owner/ref 正确 |
| AT-RESTORE-005 冲突 Skip | 目标 ConfigMap 已存在且内容不同 | policy=Skip | 目标内容不变；objectResult=Skipped 且有 diff digest；整体可 Completed（含 warning） |
| AT-RESTORE-006 冲突 Overwrite | 目标 ConfigMap 存在，用户有 overwrite 权限 | policy=Overwrite | 内容按规则更新；action=Updated；审计含 before/after digest；不删除重建无需重建的对象 |
| AT-RESTORE-007 不可变字段 Overwrite | Service clusterIP/PVC volumeName 与备份不同，allowRecreate=false | 预检/执行 | 清洗字段不强写；需要重建的冲突 `BR-RESTORE-IMMUTABLE-006` Blocking；PVC 不自动删除 |
| AT-RESTORE-008 Rename 受限 | ConfigMap 冲突且 Deployment 引用它 | policy=Rename | 创建唯一后缀 ConfigMap，并重写包内 Deployment configMapRef；对 CRD/PVC Rename 请求在预检被拒 |
| AT-RESTORE-009 PVC 恢复 | 同集群 CSI snapshot 可用、目标 SC兼容 | 新 Namespace 恢复 PVC | 必要时创建静态 VSC/目标 VS；PVC dataSource 正确并 Bound；PVC Bound 前不恢复依赖 Workload；抽样数据校验通过 |
| AT-RESTORE-010 快照丢失仅元数据 | Record=SnapshotMissing | 选择 metadataOnly/restorePVC=false | 预检允许；不创建 VS/PVC 数据恢复；资源中依赖 PVC 的 Workload按计划跳过或警告；Task结果明确 |
| AT-RESTORE-011 恢复部分失败 | 一个对象被 Webhook 拒绝，failurePolicy=Continue | 执行 | 其他无依赖对象继续；Task=PartiallyFailed；错误含 GVR/ns/name/code；失败对象导出可用 |
| AT-RESTORE-012 恢复取消 | 已创建 Namespace/PVC 后取消 | 等待终态 | 停止新 apply，Task=Cancelled；不会自动删 PVC/业务资源；residualResources 完整可导出 |
| AT-RESTORE-013 CRD 与 CR 顺序 | Record 含新 CRD 和 CR | 集群管理员恢复 | CRD 先创建并等待 Established/discovery；再恢复 CR；conversion webhook 不可达时相关 CR 不提交并明确 Blocking/Failed |
| AT-RESTORE-014 Service 清洗 | Record 中 Service 有 clusterIP/NodePort | 默认恢复 | clusterIP/clusterIPs/NodePort 按默认清除并由集群分配；启用 preserve NodePort（若实现）前冲突预检 |
| AT-RESTORE-015 恢复失败重试 | RestoreTask 因暂态 Webhook/CSI 错误 Failed，依赖已修复 | 调用 `:retry` 两次且使用同 Idempotency-Key | 系统先重新预检，只创建一个 trigger=Retry 新 RestoreTask；parentTaskRef 指向旧任务；旧任务/结果不变；新任务遵循现存目标的冲突策略并完成或给出新错误 |

## 8. 发布通过准则

1. 所有 P0 用例 100% 通过，P1 用例通过率 ≥95% 且无安全/数据损坏缺陷。
2. 无 Severity-1/2 未关闭缺陷；不存在“包未提交却 Record Available”“跨项目可见”“删除错误快照”“恢复静默覆盖”类问题。
3. 兼容矩阵中每个 Kubernetes/CSI 组合至少完成 Repo→Backup→Verify→Restore→Delete 闭环。
4. 性能压测、72 小时调度稳定性、controller 连续故障注入、SFTP/CSI 网络抖动测试均通过。
5. 生产部署文档、RBAC 清单、Secret/密钥轮换手册、灾难排查 runbook、升级/回滚说明与 Dashboard/告警规则随版本交付。
