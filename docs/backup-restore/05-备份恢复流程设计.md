# 容器平台备份与恢复插件：业务流程设计

> 图中的“API”指平台 Backup API；所有写 CRD 请求均经过 Kubernetes RBAC 和 Admission Webhook。省略的失败分支遵循：业务校验立即失败、暂态错误指数退避、任务超时进入取消补偿、所有步骤写 status/condition/event。

## 1. 创建 Repo 与连通性检查

```mermaid
sequenceDiagram
  actor U as 平台管理员
  participant UI as Console
  participant API as Platform API
  participant K as Kube API/Webhook
  participant RC as RepoController
  participant S as Secret/PVC/Node
  participant A as RepositoryAdapter
  U->>UI: 填写 Local/SFTP 与 SecretRef
  UI->>API: POST /repositories
  API->>API: repo.manage + 字段/Secret元数据校验
  API->>K: Create BackupRepository
  K->>K: Webhook 校验路径/引用/不可变规则
  K-->>UI: 202 Pending
  K-->>RC: Watch event
  RC->>S: 读取所需 Secret/PVC/Node（不记录内容）
  RC->>A: DNS/TCP/认证/路径/RW/delete/rename/statfs
  alt 全部通过
    A-->>RC: capabilities + capacity
    RC->>K: status Ready, Ready=True
  else 可用但不满足建议
    RC->>K: status Degraded + warning
  else 不可用
    RC->>K: status Unavailable + errorCode
  end
  UI->>API: GET /repositories/{name}
  API-->>UI: 脱敏状态与分步检查结果
```

规则：测试文件位于 `.health/<repoUID>/<uuid>`，即使失败也在 finally 清理；测试成功不是永久保证，按 interval 复检。SFTP host key 不匹配直接 `BR-REPO-HOSTKEY-003`，不允许自动接受新 key。

## 2. 在 BackupPolicy 中配置范围并预览

```mermaid
sequenceDiagram
  actor U as 管理员
  participant UI as Console
  participant API as Platform API
  participant DIR as Project Directory
  participant PC as PolicyController/Preview
  participant K as Kubernetes API
  participant CSI as CSI Discovery
  U->>UI: 填写策略名称并选择 Namespace/GVR/标签/PVC/Repo
  UI->>API: POST /policies (enabled=false)
  API->>DIR: 获取 allowedNamespaces + authzVersion
  API->>K: Create BackupPolicy
  K-->>PC: Watch Policy
  PC->>K: API Discovery + SSAR
  loop 分页资源
    PC->>K: List GVR namespace selector limit=500
    K-->>PC: items + continue
  end
  PC->>CSI: 检查 PVC→PV→driver→VSC
  PC->>K: status.selectionPreview counts/warnings/hash
  API-->>UI: 策略详情与选择预览
  U->>UI: 核对后启用策略
  UI->>API: POST /policies/{name}:enable
  API->>K: Patch BackupPolicy enabled=true
  K-->>UI: Policy Ready/Invalid
```

策略选择配置只有一个事实源：`BackupPolicy.spec.selection`。Controller 在策略 generation 变化时立即刷新预览，之后最长每 10 分钟刷新一次；预览不返回未授权对象名称。执行任务创建时会把当前选择冻结到 `BackupTask.spec.selectionSnapshot`。

## 3. 校验并启用 BackupPolicy

```mermaid
sequenceDiagram
  actor U as 管理员
  participant UI as Console
  participant API as Platform API
  participant K as Kube API/Webhook
  participant PC as PolicyController
  participant R as Repo/Selection Cache
  U->>UI: 核对 Selection 预览/Repo/Cron/保留
  UI->>API: POST /policies:validate
  API-->>UI: 未来5次执行/DST/引用状态
  U->>UI: 启用
  UI->>API: POST /policies/{name}:enable
  API->>K: Patch BackupPolicy
  K->>K: 校验同 cluster/project、Cron/timezone、权限
  K-->>PC: Watch
  PC->>R: 解析 name→UID/Ready
  PC->>PC: 计算 scheduleHash/nextRunTime
  PC->>K: status Ready, nextRunTime
  K-->>UI: 200 Policy
```

创建默认 enabled=false；启用不会立即备份。立即执行必须走单独 action。

## 4. 定时生成 BackupTask

```mermaid
sequenceDiagram
  participant Clock as Requeue/Clock
  participant PC as PolicyController Leader
  participant K as Kubernetes API
  participant IDX as Task Index
  Clock->>PC: 到 nextRunTime / 重启补偿
  PC->>K: 读取 Policy/Scope/Repo/Config 当前状态
  PC->>PC: 计算 missed points + concurrency policy
  PC->>IDX: 查询 policyUID + scheduledTime
  alt 已存在
    IDX-->>PC: existing Task
    PC->>K: 更新 lastEvaluated/nextRun（不重复）
  else Forbid 且有活动任务
    PC->>K: skippedRuns += ForbidActive
  else Replace
    PC->>K: patch oldTask cancelRequested=true
    PC->>PC: 等待终态/宽限期
  else 允许创建
    PC->>K: Create BackupTask deterministic-name
    Note over PC,K: idempotencyKey=policyUID/scheduledTimeUTC
    K-->>PC: Created 或 AlreadyExists
    PC->>K: 更新 lastScheduleTime/nextRunTime
  end
```

控制器停机后：Skip 不补；RunOnce 取 startingDeadline 内最近点；RunAll 从旧到新最多 10。Task spec 固化 Policy UID/generation、Selection、Repo UID/generation，策略后续修改不影响它。

## 5. 手动创建 BackupTask

```mermaid
sequenceDiagram
  actor U as 授权用户
  participant UI as Console/CLI
  participant API as Platform API
  participant DIR as IAM/Project
  participant K as Kube API/Webhook
  participant BTC as BackupTaskController
  U->>UI: 选择策略
  UI->>API: POST /backup-tasks + Idempotency-Key
  API->>DIR: backup.execute + Namespace 授权
  API->>K: Create BackupTask trigger=Manual, policyRef
  alt 同幂等键已存在
    K-->>API: existing Task
    API-->>UI: 200 same task
  else 新任务
    K-->>BTC: Watch event
    BTC->>K: 一次性解析并固化 Selection/Repo/Policy generation
    API-->>UI: 202 + task URL
  end
  BTC->>K: Pending→Validating
```

手动 Task 不修改/消费 Policy 下次执行时间。基于 Policy 立即执行时保留 policyRef，但 trigger=Manual。

## 6. 资源对象采集

```mermaid
sequenceDiagram
  participant BTC as BackupTaskController
  participant W as Backup Worker
  participant K as Kubernetes API
  participant F as Filter/Sanitizer
  participant WS as Workspace
  BTC->>W: 启动确定名称 Job + selectionSnapshot
  W->>K: Discovery + SelfSubjectAccessReview
  loop 每个 GVR/Namespace 分页
    W->>K: LIST labelSelector limit=500
    K-->>W: items/resourceVersion/continue
    W->>F: 授权交集→include/exclude→系统排除
    F->>F: 删除 status/runtime metadata/finalizer
    F->>WS: 流式写 object + dependency/source UID index
    W->>BTC: 节流更新 processed/success/failed
  end
  W->>WS: 完成稳定排序 index
  W-->>BTC: Collection checkpoint + counts/hash
```

采集不是 API Server 一致性快照：每 GVR 记录 list resourceVersion 和采集窗口；metadata.json 明示 `consistency=BestEffort`。资源在采集期间变化产生 warning；V1.0 不通过 etcd snapshot 保证跨 GVR 原子一致性。

## 7. PVC 快照创建

```mermaid
sequenceDiagram
  participant W as Backup Worker
  participant SM as SnapshotManager
  participant K as Kubernetes API
  participant C as snapshot-controller
  participant D as CSI Driver
  W->>SM: PVC plan + class mapping + timeout
  loop 每个 PVC（并发受限）
    SM->>K: GET PVC/PV/SC/VolumeSnapshotClass
    SM->>K: CREATE/GET VolumeSnapshot deterministic-name
    K-->>C: Watch VolumeSnapshot
    C->>D: CreateSnapshot
    D-->>C: snapshotHandle
    C->>K: bind VolumeSnapshotContent, readyToUse=true
    SM->>K: Watch/relist until readyToUse
    alt 成功
      SM-->>W: PVC→VS/VSC/handle/class/driver/lifecycle
    else timeout/unsupported
      SM-->>W: errorCode + PVC
      W->>W: Fail 或 partial（按 failurePolicy）
    end
  end
```

只读取 handle 存入加密包，不在普通 UI 展示。Task 删除不删除已提交 Record 的快照；未提交任务失败时，只有 Controller 创建且生命周期允许的快照才补偿删除。

## 8. 备份包归档和上传

```mermaid
sequenceDiagram
  participant W as Worker
  participant AM as ArchiveManager
  participant CM as ChecksumManager
  participant RA as RepositoryAdapter
  participant REPO as Local/SFTP
  W->>AM: manifests/index/snapshot results/logs
  AM->>AM: metadata.json + index.json + snapshots.json
  AM->>AM: tar stream→gzip→AES-GCM（可选）
  AM->>CM: 对最终存储字节计算 SHA-256/size
  CM-->>W: sha256sum.txt + manifest checksum
  W->>RA: Begin(backupID)
  RA->>REPO: mkdir .staging/backupID
  loop 每个文件
    RA->>REPO: write file.part + fsync/close
    W->>W: status bytesUploaded/checkpoint
  end
  RA->>REPO: rename staging/定名文件
  RA->>REPO: 最后写 .done
  RA-->>W: commit marker confirmed
  W->>W: phase Verifying
```

上传中无 `.done` 的目录永不被视为副本。SFTP rename 不支持时逐文件提交，仍以 `.done` 为可见性屏障。路径组件由 backupID 生成并防 `..`/symlink 逃逸。

## 9. 生成 BackupRecord

```mermaid
sequenceDiagram
  participant BTC as BackupTaskController
  participant RA as RepositoryAdapter
  participant K as Kubernetes API
  participant BRC as BackupRecordController
  BTC->>RA: 重新打开 .done/manifest/files
  RA-->>BTC: 全量 hash/size 正确 + snapshot refs
  BTC->>K: Create BackupRecord deterministic-name
  alt 已存在且 backupID/taskUID 匹配
    K-->>BTC: adopt existing
  else 名称冲突但身份不匹配
    K-->>BTC: integrity conflict
    BTC->>K: Task Failed（不覆盖）
  else 新建成功
    K-->>BRC: Record event
    BRC->>RA: 快速确认 .done + manifest
    BRC->>K: availability=Available/ SnapshotMissing
    BTC->>K: task.recordRef + Completed/PartiallyFailed
  end
```

顺序不可倒置：commit → verify → Record create → Record accepted → Task terminal。若 Record 创建暂时失败，Task 留在 GeneratingRecord 重试，不重复上传。

## 10. 备份完整性校验

```mermaid
sequenceDiagram
  actor U as 管理员/周期调度
  participant API as Platform API
  participant K as Kubernetes API
  participant BRC as BackupRecordController
  participant RA as RepositoryAdapter
  participant CSI as Snapshot API
  U->>API: POST record:verify（或到周期）
  API->>K: 写 verify request nonce
  K-->>BRC: Reconcile
  BRC->>K: availability=Verifying
  BRC->>RA: GET .done, metadata,index,sha256sum
  loop 声明文件
    BRC->>RA: stat + stream SHA-256
  end
  BRC->>CSI: 检查 VS/VSC/handle 存在
  alt 包和快照完整
    BRC->>K: Available + IntegrityVerified=True
  else 包损坏
    BRC->>K: Broken + errorCode
  else 快照缺失
    BRC->>K: SnapshotMissing + missing list
  else Repo 不可访问
    BRC->>K: RepoUnavailable（不判 Broken）
  end
```

校验锁保证同 Record 仅一个；恢复开始时要求校验在全局 freshness 内，否则先同步快速校验。校验不自动“修复”包。

## 11. 创建 RestoreTask

```mermaid
sequenceDiagram
  actor U as 恢复用户
  participant UI as Restore Wizard
  participant API as Platform API
  participant K as Kubernetes API/Webhook
  participant RP as RestorePlanner
  participant RTC as RestoreTaskController
  U->>UI: 选择 Record/目标/映射/资源/PVC/冲突
  UI->>API: POST /restore-plans
  API->>RP: 生成计划（不写业务资源）
  RP-->>UI: blocking/warnings/diff/planHash/TTL
  U->>UI: 确认 warning/高危操作
  UI->>API: POST /restore-tasks + planHash
  API->>API: restore.create + 二次确认 token
  API->>K: Create RestoreTask
  K->>K: 校验 Record UID/权限/目标/plan TTL
  K-->>RTC: Watch event
  API-->>UI: 202 task URL
```

V1.0 targetClusterRef 必须等于 Record source cluster。任何 Namespace 授权、Record checksum、计划输入或 discovery hash 变化都会使 plan stale。

## 12. 恢复预检查

```mermaid
sequenceDiagram
  participant RTC as RestoreTaskController
  participant RA as RepositoryAdapter
  participant K as Target Kubernetes API
  participant IAM as AuthZ/SSAR
  participant CSI as CSI/Snapshot
  participant WH as Admission/Conversion Webhooks
  RTC->>RA: 校验/下载 metadata,index,所需资源
  RTC->>K: Discovery + cluster UID/version
  RTC->>IAM: 对目标 GVR/Namespace 批量 SSAR
  RTC->>K: Namespace/ResourceQuota/LimitRange/冲突 GET
  RTC->>CSI: SC/driver/class/handle/容量检查
  RTC->>WH: 可行时 server-side dry-run 探测
  RTC->>K: CRD conversion webhook service/endpoints 检查
  RTC->>RTC: 依赖 DAG + 冲突 action + 不可变字段判定
  alt blocking > 0
    RTC->>K: Failed, BR-RESTORE-PRECHECK-...
  else dryRun
    RTC->>K: Completed + plan report（不写资源）
  else 可执行
    RTC->>K: PreCheckPassed=True → RestoringNamespaces
  end
```

预检查通过不是永久保证，执行每个对象前仍处理竞态冲突。Webhook 不支持 dry-run 时标 Warning，不绕过 webhook。

## 13. PVC 恢复

```mermaid
sequenceDiagram
  participant RTC as Restore Controller/Worker
  participant K as Kubernetes API
  participant C as snapshot-controller
  participant D as CSI Driver
  RTC->>K: 确认目标 Namespace 已就绪
  alt 原 Namespace 且源 VolumeSnapshot 可引用
    RTC->>K: GET/adopt source VolumeSnapshot
  else 新 Namespace
    RTC->>K: CREATE static VolumeSnapshotContent from snapshotHandle (Retain)
    RTC->>K: CREATE target Namespace VolumeSnapshot bound to content
  end
  RTC->>K: CREATE PVC with dataSource VolumeSnapshot，清 volumeName/binding annotations
  K-->>C: reconcile restore
  C->>D: CreateVolume from snapshot
  D-->>K: PV provisioned / PVC Bound
  RTC->>K: Watch PVC until Bound/timeout
  alt Bound
    RTC->>RTC: 写 PVC mapping/checkpoint
  else driver/handle/SC/配额失败
    RTC->>RTC: Fail 或 partial；依赖 Workload 不恢复
  end
```

`WaitForFirstConsumer` 的 StorageClass 可能在无 Pod 时不绑定：V1.0 预检查提示，可先创建受控 placeholder consumer 或延迟到 Workload 组；默认采用延迟恢复 Workload 并等待调度，需存储厂商认证。

## 14. Kubernetes 资源恢复

```mermaid
sequenceDiagram
  participant RTC as Restore Controller
  participant RR as ResourceRestorer
  participant K as Kubernetes API
  participant CR as ConflictResolver
  loop 按依赖组与稳定顺序
    RTC->>RR: next source object + UID map
    RR->>RR: 清 status/runtime/finalizer，重写 namespace/ref
    RR->>K: GET target
    alt 不存在
      RR->>K: CREATE + restore annotations
      K-->>RR: new UID
    else 已存在
      RR->>CR: source/target/policy
      alt Skip
        CR-->>RR: record skipped+diff
      else Fail
        CR-->>RTC: object failed/block
      else Overwrite
        RR->>K: SSA/update 或受控 recreate
      else Rename
        RR->>RR: 生成名称并重写已知引用
        RR->>K: CREATE renamed object
      end
    end
    RR->>RTC: checkpoint action/result/new UID
  end
  RTC->>K: 二次 patch ownerReferences with new UIDs
  RTC->>K: GET 验证对象/PVC
  RTC->>K: status Completed/PartiallyFailed/Failed
```

固定组序：Namespace、CRD、集群依赖、ServiceAccount、Secret、ConfigMap、RBAC、PVC、Service、Workload、Ingress、HPA/PDB/NetworkPolicy、CR。Service 默认不保留 clusterIP/NodePort；PVC 不恢复 PV 绑定；源 finalizer 默认不恢复。

## 15. 删除 BackupRecord

```mermaid
sequenceDiagram
  actor U as 授权管理员
  participant UI as Console
  participant API as Platform API
  participant K as Kube API/Webhook
  participant BRC as BackupRecordController
  participant RA as RepositoryAdapter
  participant CSI as Snapshot API/Driver
  U->>UI: 选择 CROnly/Data/DataAndSnapshots
  UI->>API: POST record:delete + recordName confirmation
  API->>API: 权限/引用中恢复任务/确认 token/影响预检
  API->>K: 写签名 deletion request + DELETE Record
  K->>K: Webhook 校验 5m token/mode
  K-->>BRC: deletionTimestamp + finalizer
  BRC->>K: availability=Deleting
  alt CROnly
    BRC->>BRC: 跳过外部资产，记录 orphan audit
  else Data 或 DataAndSnapshots
    BRC->>RA: delete package files，最后目录/.done
    RA-->>BRC: absent confirmed
    opt DataAndSnapshots
      loop 每个受管快照
        BRC->>CSI: delete VS/VSC/underlying per ownership
        CSI-->>BRC: absent confirmed
      end
    end
  end
  BRC->>K: status deletion checkpoints
  BRC->>K: remove finalizer
  K-->>UI: Record deleted
```

有活动 RestoreTask 引用时禁止删除。Repo 不可访问时保留 finalizer 并可重试；平台管理员可在风险审批后强制 abandon finalizer，此动作产生不可篡改审计和孤儿资产报告。

## 16. 保留策略与垃圾回收

```mermaid
sequenceDiagram
  participant Tick as Periodic Tick
  participant RET as RetentionController
  participant K as Kubernetes API
  participant BRC as RecordController
  participant GC as GarbageCollectionController
  participant RA as Repo
  participant CSI as Snapshot API
  Tick->>RET: retention interval
  RET->>K: List Policy/Record by policyUID
  RET->>RET: 排除 protected/active restore/minRecords
  RET->>RET: 按 age/count 选最旧候选
  loop 候选 Record
    RET->>K: 写受签名 delete request + mode
    K-->>BRC: deletion reconcile
    BRC->>RA: 删除包
    opt policy.retention.deleteSnapshots
      BRC->>CSI: 删除受管快照
    end
    BRC->>K: remove finalizer
  end
  Tick->>GC: GC interval
  GC->>K: 读取活动/终态 Task/Record 索引
  GC->>RA: list .staging 与 package headers
  GC->>GC: ownership + grace + no active task 三重判断
  GC->>RA: 删除确定孤儿 staging
  GC->>K: 删除过期内部 Job/Pod；未知资产仅报告
  GC->>K: 写 GC report/Event/metrics
```

Retention 选择规则：保护标签/法律保留优先；至少保留 minRecords；到期或超过 maxRecords 的最旧记录成为候选。删除 Policy 不触发 Retention 级联；失去 Policy 的 Record 仍按自身 `expiresAt` 由全局 retention 处理。GC 永不根据“目录名看起来像备份”删除未知数据。

## 17. 异常与补偿总则

| 异常点 | 资产状态 | 补偿/恢复 |
|---|---|---|
| 采集中断 | 无 Repo commit；可能有 workspace | 重建 Job 后重新采集；GC 清 workspace |
| 部分快照创建后失败 | 未形成 Record 的受管快照 | Fail 时按 lifecycle 清理；无法清理列 orphan；Continue 可写 partial Record |
| 上传中断 | `.staging` 无 `.done` | timeout 内重试；终态后 grace 24h 由 GC 清理 |
| `.done` 已写、Record 未建 | 完整孤儿包 | Task 从 checkpoint 生成 Record；GC 识别 active Task，不清理 |
| checksum mismatch | Record 不创建或置 Broken | 禁止恢复；保留证据，人工排查存储/传输 |
| 恢复中取消 | 已创建资源可能存在 | 不自动回滚业务数据；输出 residual resources 和 action 日志 |
| Webhook 临时不可用 | 对象 apply 失败 | 瞬态重试至 timeout；Continue 可部分失败 |
| CRD conversion webhook 永久不可用 | 相关 CR 无法可靠转换 | 预检查 Blocking；先恢复/修复 webhook 依赖 |
| Repo 删除/凭据轮换 | Record 可能 RepoUnavailable | 新凭据重检；不把网络错误判为损坏 |
